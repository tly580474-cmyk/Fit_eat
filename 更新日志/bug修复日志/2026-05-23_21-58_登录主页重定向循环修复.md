# Bug 修复日志

**日期时间：** 2026-05-23 21:58

## Bug 描述

重新初始化数据库后，login.html 和 home.html 之间出现无限重定向循环，控制台报 `ERR_ABORTED` 错误。

## 根本原因

1. `login.html` 只检查 localStorage 中是否有 token，不验证 token 是否有效，直接跳转 home
2. `home.html` 的 `initPage()` catch 块仅打印错误日志，未清理无效 token 或重定向
3. `api.js` 的 `request` 函数未全局处理 401 响应

三个问题叠加形成死循环：login 检测到 token → 跳 home → API 失败 → 未清理 token → 回到 login → 又检测到 token → 循环

## 修复方案

### 1. `js/api.js` — 全局 401 处理

在 `request()` 函数中增加 401 状态码检测，自动清理 token 并跳转登录页：

```javascript
if (res.status === 401) {
  localStorage.removeItem('token');
  localStorage.removeItem('accountId');
  if (currentPage !== 'login.html' && currentPage !== 'register.html') {
    window.location.href = 'login.html';
  }
  return { success: false, message: '未登录或登录已过期' };
}
```

### 2. `pages/home.html` — catch 块增加清理逻辑

在 `initPage()` 的 catch 块中清理 localStorage 并跳转登录页：

```javascript
} catch (error) {
  console.error('加载数据失败:', error);
  localStorage.removeItem('token');
  localStorage.removeItem('accountId');
  window.location.href = 'login.html';
}
```

### 3. `pages/login.html` — 跳转前验证 token

将简单的 `if (localStorage.getItem('token'))` 改为先调用 `API.user.getProfile()` 验证 token 有效性，无效则清理 token 留在登录页：

```javascript
if (localStorage.getItem('token')) {
  API.user.getProfile().then(res => {
    if (res && res.success !== false && res.id) {
      window.location.href = 'home.html';
    } else {
      localStorage.removeItem('token');
      localStorage.removeItem('accountId');
    }
  }).catch(() => {
    localStorage.removeItem('token');
    localStorage.removeItem('accountId');
  });
}
```

## 修改文件

| 文件 | 修改内容 |
|------|----------|
| `轻饮食软件前端/js/api.js` | request() 增加全局 401 处理 |
| `轻饮食软件前端/pages/home.html` | initPage() catch 块增加 token 清理和跳转 |
| `轻饮食软件前端/pages/login.html` | 跳转前先验证 token 有效性 |

## 防护层级

本次修复建立了三层防护机制：
1. **API 层**：任何 401 响应自动触发清理和跳转
2. **页面层**：网络错误等异常情况也触发清理和跳转
3. **入口层**：登录页跳转前先验证 token，避免无效跳转
